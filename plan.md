# GitHub 初心者向け 2 時間ワークショップ構成案（社内 FAQ/ヘルプデスク刷新ストーリー）

## ワークショップのゴール（初心者向け必須ポイント）

- GitHub Desktop / Web だけで「変更→コミット→プッシュ」ができる。
- Issue からブランチを作り、Pull Request を出してレビュー→マージまでの最短ルートを体験する。
- GitHub Projects で自分のタスク状況（未着手/レビュー待ち）を動かせる。
- Copilot を使って FAQ 文書の下書きやレビューコメントを試し、どこまで頼れるかを知る。
- 2 時間で最低 1 つの FAQ を更新し、実務に直結する成果物を持ち帰る。

## ストーリー概要

情シス部門「HelpDesk Motor Company」に新任として参加し、滞留している問い合わせ（VPN、Teams、プリンタなど）を 2 時間で FAQ に反映させるミッション。操作は GitHub Desktop / Web UI を中心にし、「変更→レビュー→公開」をシンプルに一周する。

## 事前準備（参加者向け案内）

- GitHub アカウント（2FA 済）と GitHub Desktop、VS Code（Copilot 拡張入り）。CLI は不要。
- 配布リポジトリ `helpdesk-faq-sample` をフォークし、GitHub Desktop で clone しておく。
- リポジトリに含まれる主要フォルダだけ事前に眺めておく：`docs/faq/`（FAQ 原稿）、`.github/ISSUE_TEMPLATE/`（問い合わせテンプレ）、`projects/`（ボード定義）。
- Codespaces or 事前構築 VM を利用する人は、VS Code から動くことを確認。

## セッション詳細

### 1. オープニング（10 分）

- アイスブレイク：最近社内で起きた IT トラブルを 1 分で共有し、ボードに貼る。
- HelpDesk Motor Company のストーリーと「2 時間で FAQ を 1 件以上改善する」というゴールを共有。
- 役割カード（一次受け／編集／レビュー／公開）を配布し、区切りごとに交代するルールだけ確認。

### 2. GitHub Desktop 基本操作（20 分）

- デモ：GitHub Desktop で clone → 変更チェック → 「Changes」でステージング → コミット → 「Push origin」まで GUI で完了。
- 演習：VS Code で誤植を 1 か所直し、Desktop の Changes 画面だけを使ってコミット＆同期。
- 3 分チャレンジ：Issue 番号入りメッセージで push まで終えた人にバッジ。
- 補足：VS Code Source Control の「Discard」で元に戻せることだけ紹介（コマンドは後述資料）。

### 3. Issue 化とブランチ作成（20 分）

- Web UI で問い合わせテンプレを開き、タイトル・再現手順・優先度を埋めて Issue を作成。
- Issue 右上の「Create branch」ボタンで `issue-123-update-vpn-guide` を自動作成し、名前の付け方を確認。
- GitHub Desktop の Branch メニューで新しいブランチを選択し、VS Code で FAQ を編集開始。
- ブランチのイメージ（枝分かれ→main に戻す）を簡易図で共有。CLI 例は配布資料に記載するだけ。

### 4. GitHub Projects で進行可視化（15 分）

- ボードの列（未着手/作業中/レビュー/完了）の意味を確認し、自分の Issue カードをドラッグ。
- 優先度と期限をカスタムフィールドで入力し、色ラベルがどう反映されるかを見る。
- このセッションで覚えることは「カードを動かす」「フィールドを編集する」の 2 つだけ。CLI 例は紹介に留める。
- 時間が余れば、ユーザーストーリーを 1 文でまとめてカードに追記。

### 5. Pull Request & GUI ワークフロー（20 分）

- GitHub Desktop で差分プレビューと「Open in VS Code」を見せ、レビュー前に内容を確認する流れを把握。
- Push 後に表示される「Compare & pull request」から PR を作成し、テンプレの 3 項目（背景・変更点・確認方法）だけ記入。
- VS Code Source Control で `docs/faq/*.md` の差分を眺め、GUI だけでレビュー準備ができることを体感。
- CLI での PR 作成は「興味がある人は後で試してね」と軽く触れる程度にとどめる。

### 6. レビュー & Copilot 応用（20 分）

- Copilot Chat に Issue 内容を貼り、「FAQ 文章にリライトして」とお願いしてドラフトを得る。
- PR 画面で `suggested change` を 1 回使ってもらい、GUI だけで修正提案→適用まで進める。
- Copilot Code Review（ボタン 1 つで自動コメント）を流し、指摘の読み方と対応の仕方を共有。
- 承認ボタンを押す前に「背景/変更点/確認方法」を声に出してチェックし、レビューの基本を定着させる。

### 7. マージ後の成果可視化（10 分）

- PR 画面の「Merge pull request」→「Confirm merge」を押し、main に入る瞬間を全員で確認。
- GitHub Actions が README を更新し、最新 FAQ リストが変わる様子をダッシュボードで見る。
- 公開ページ（もしくはリポジトリ内の HTML プレビュー）が更新されたことを確認し、SLA メーターを減らしてゴールを可視化。コマンドでのマージは参考資料に記載。

### 8. クロージング（5 分）

- KPT フォーマットで気づきを共有（Keep: Copilot 便利、Problem: CLI がまだ怖い 等）。
- 次の一歩（ブランチ保護ルール、Issue テンプレ高度化、Actions で FAQ 自動配信）のアイデアを提示。
- フィードバックフォームと「今日のベスト Tips」投票で締め、集合写真/スクリーンショットを撮影。

## 参加者をワクワクさせる仕掛け

- **役割カード & ローテーション**: 一次受け/検証/レビュー/公開の役割を入れ替え、ストーリー没入感を演出。
- **問い合わせバッジチャレンジ**: GUI 操作でのコミット/PR 作成、緊急 Issue 解決、Copilot 連携などでシールを獲得し、最後に「SLA Champion」を称える（CLI 達成はオプション）。
- **リアルタイム SLA メーター**: 残り問い合わせ数をスライドに表示し、PR マージでゲージが下がる仕掛けで進捗を視覚化。
- **質問パーキングボード**: 不明点を付箋で貼り、クロージングでまとめて回答して心理的安全性を確保。
- **フォトタイム**: 更新済み FAQ とチーム写真を撮影し、社内ポータルに掲載できる素材として配布。

## 実施上のヒント

- 参加者 2-3 名ごとにメンター（TA）を配置し、詰まりを即時解消。
- 各セッション終わりに「ここまでできたら OK」チェックリストを提示し、遅れがちな人をフォロー。
- ハンズオン資料（スライド + README + GUI 操作早見表 + 参考コマンド集）を事前共有し、コピー＆ペーストで進められる安心感を提供。
- Codespaces や事前構築済み VM を用意し、環境トラブルを最小化。
- 進行タイマーと進捗メーターをスクリーンに表示して集中力を維持。
- ショートブレイクで GitHub モバイルや Copilot Labs の Tips を紹介し、ヘルプデスク運用の幅を感じてもらう。
- ワークショップ後に録画・FAQ・追加課題（例: 「FAQ を 1 つ自動化する Actions を作る」）を Issue で配布し、現場適用まで伴走する。

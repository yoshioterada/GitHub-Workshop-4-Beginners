# 進行スクリプト（Facilitation Script）

HelpDesk Motor Company FAQ 刷新ワークショップを 120 分間スムーズに進めるための台本です。時間配分、声掛け例、トラブル対応をまとめました。司会/進行役・サポートメンターで共有してください。

---

## タイムテーブル概要

| 時刻 | セッション | 進行ポイント |
| --- | --- | --- |
| 0:00-0:10 | オープニング | ストーリー導入、ゴール説明、役割カード配布 |
| 0:10-0:30 | GitHub Desktop 基本操作 | クローン、ステージ、コミット、プッシュを GUI で実演 |
| 0:30-0:50 | Issue 化 & ブランチ作成 | テンプレ入力→ブランチ発行→VS Code 編集開始 |
| 0:50-1:05 | GitHub Projects 実演 | ボード列移動、カスタムフィールド更新 |
| 1:05-1:25 | Pull Request 作成 | PR テンプレ、差分確認、Copilot サジェスト紹介 |
| 1:25-1:45 | レビュー & Copilot 応用 | Chat / Code Review、suggested changes |
| 1:45-1:55 | マージ & 成果確認 | Actions 成功、Pages 更新、SLA メーター更新 |
| 1:55-2:00 | クロージング | KPT、フィードバック、記念撮影 |

---

## セッション別詳細台本

### 0:00-0:10 オープニング

- **スライド**: HelpDesk Motor Company のストーリー、FAQ 滞留状況。
- **声掛け例**: 「今日は 2 時間で FAQ を 1 件以上改善し、ヘルプデスクの SLA を回復させます！」
- **役割カード**: 一次受け / 編集 / レビュー / 公開 を配布。セッションごとに交代するルールを説明。
- **トラブル対応**: 参加者が座席/設備で困っていたらメンターを呼ぶ。

### 0:10-0:30 GitHub Desktop 基本操作

- **デモ手順**:
  1. GitHub Desktop でリポジトリを clone。
  2. 誤字修正 → コミット → Push。
- **声掛け**: 「3 分チャレンジ！Issue 番号入りメッセージで push まで行ったらバッジです。」
- **トラブル対応**: clone できない場合は zip ダウンロードか Codespaces へ誘導。

### 0:30-0:50 Issue 化 & ブランチ作成

- **手順**:
  1. ブラウザで Issue テンプレ `Helpdesk Ticket` を開く。
  2. `Create branch` を押して `issue-###-短い説明` を自動作成。
  3. GitHub Desktop でブランチ切替、VS Code で該当 FAQ を開く。
- **声掛け**: 「テンプレの SLA 欄は必ず埋めてください。Projects に自動でカードが乗ります。」
- **トラブル対応**: ブランチが見つからない→`Fetch origin` / `Branch > Checkout` を案内。

### 0:50-1:05 GitHub Projects で可視化

- **デモ**: `projects/helpdesk-refresh.json` の列（未着手/作業中/レビュー/公開済み）を紹介。
- **ワーク**: 参加者の Issue カードをドラッグ＆ドロップで「作業中」へ移動。
- **声掛け**: 「カードを動かす × 優先度/期限を入力する、この 2 点だけ覚えれば OK」
- **トラブル対応**: カードが見つからない→検索バーでタイトルを入れる。

### 1:05-1:25 Pull Request 作成

- **手順**:
  1. GitHub Desktop で差分を確認。
  2. Push 後の「Create Pull Request」からテンプレを埋める。
  3. Copilot Chat で文章リライトを体験。
- **声掛け**: 「背景・変更点・確認方法を 1 行でも良いので埋めましょう。Copilot に 'レビューコメント案を作って' と頼んでも OK。」
- **トラブル対応**: ボタンが出ない→Push し忘れか、既存 PR があるので確認。

### 1:25-1:45 レビュー & Copilot 応用

- **手順**:
  1. ペアで PR を開き、`Files changed` から `Add comment`/`Add suggestion`。
  2. Copilot Code Review を流し、自動コメントの読み方を解説。
  3. 指摘を反映→再コミット→再レビュー。
- **声掛け**: 「コメントはポジティブに。『背景/変更点/確認方法』を声に出して確認してから Approve。」
- **トラブル対応**: Copilot が動かない→`Help > GitHub Copilot > Sign in` を案内、企業ポリシー要確認。

### 1:45-1:55 マージ & 成果可視化

- **手順**:
  1. `Merge pull request` → `Confirm merge`。
  2. Actions `Update README` の成功を確認。
  3. GitHub Pages or README の FAQ 一覧が更新されたことを示す。
  4. Projects カードを「公開済み」へ移動。
- **声掛け**: 「SLA メーターを 1 つ減らしましょう！」
- **トラブル対応**: コンフリクト発生時は進行役が CLI で解消 or 作業内容を手動マージ。

### 1:55-2:00 クロージング

- KPT（Keep/Problem/Try）を付箋やオンラインボードで回収。
- フィードバックフォーム / ベスト Tips 投票の案内。
- 集合写真 or スクリーンショット撮影。

---

## トラブルシュート集

| 状況 | 対応策 |
| --- | --- |
| GitHub Desktop でログインできない | ブラウザ連携 (`File > Options > Accounts > Sign in`) を案内。必要なら PAT を作成。 |
| Push ができない | `git status` / `git pull --rebase origin main`（CLI）をメンターが実行し調整。 |
| Copilot ライセンスなし | 代替案として ChatGPT/社内ツールを紹介し、後日手順を共有。 |
| Actions が失敗 | 進行役が `docs/actions-verification.md` の手順で再実行、ログを参加者へ画面共有。 |
| ネットワーク不調 | Codespaces や事前配布された VM に切り替え、資料のオフライン版を渡す。 |

---

## 声掛けテンプレート

- **進捗確認**: 「いまの操作で詰まっている方いますか？手を挙げてください。」
- **フォロー**: 「コミットメッセージに Issue 番号（例: `Refs #101`）を入れると後から追跡しやすいです。」
- **盛り上げ**: 「次に PR を出せたチームには『SLA Champion』バッジを進呈します！」
- **時間管理**: 「残り 5 分で PR 作成パートを締めます。まだコミットできていない方はメンターが手伝います。」

## 進行役の持ち物リスト

- スライド / タイムキーパー用デバイス
- GitHub Projects の大画面表示用ブラウザ
- バッジや景品（任意）
- 予備 PC / テザリング端末
- 連絡手段: Slack `#helpdesk-live` チャンネル

このスクリプトを `docs/workshop-operation.md` と併せて読み、担当メンター全員で共有してください。状況に応じて柔軟にアレンジしつつ、時間配分と声掛けを意識することで、コンフリクトの少ない円滑なワークショップ運営が実現できます。
